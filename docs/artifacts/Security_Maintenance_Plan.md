*사이버 보안 유지보수 계획서 (Security Maintenance Plan)*

*제품*: Simple CBCT Viewer
*문서 ID*: SMP-PLAYG-1233
*작성일*: 2026-04-09
*대상 티켓*: PLAYG-1233
*IEC 62304 안전 등급*: Class A (PLAYG-1290 Classification 참조)
*적용 규격*: IEC 62304:2006+AMD1:2015, ISO 14971:2019

---

*1. [SMP-01] 목적 및 적용 범위 (Post-market Cybersecurity)*

*1.1 목적*
본 문서는 Simple CBCT Viewer 소프트웨어의 출시 후(post-market) 사이버 보안 유지보수 활동을 계획하고 관리하기 위한 사이버 보안 유지보수 계획서(Security Maintenance Plan)이다. IEC 62304:2006+AMD1:2015 제6절(소프트웨어 유지보수 프로세스) 및 FDA 사이버 보안 가이던스에 따라 출시 후 사이버 보안 활동을 정의한다.

*1.2 적용 대상 소프트웨어*
- 제품명: Simple CBCT Viewer
- 유형: 웹 브라우저 기반 CBCT 영상 뷰어 (독립형 소프트웨어, 오프라인 동작)
- 의도된 목적: 치과용 CBCT 영상의 조회 및 보조 분석 (진단 목적 아님)
- Intended Use 참조: PLAYG-1237
- Classification 참조: PLAYG-1290 (Class A)

*1.3 적용 범위*
- 본 계획은 Simple CBCT Viewer의 상용 출시 이후 전체 수명 주기 동안 적용된다.
- 주요 활동: 취약점 모니터링, 패치 관리, 정보 제공, 사고 대응
- 본 소프트웨어는 오프라인 로컬 동작(SR-7)을 원칙으로 하므로 네트워크 기반 공격 표면이 제한적이나, DICOM 파일 입력 처리, 브라우저 렌더링 엔진 의존성 등으로 인한 사이버 보안 위험을 관리한다.

*1.4 관련 문서*
- Intended Use: PLAYG-1237
- System Requirement Specification: PLAYG-1229 (SyRS SR-1 ~ SR-12)
- Software Development Plan: PLAYG-1231 (DP-PLAYG-1231)
- Risk Management Plan: PLAYG-1232 (RMP-PLAYG-1232)
- Classification: PLAYG-1290 (Class A)
- Configuration Management Plan: PLAYG-1234

*2. [SMP-02] 사이버 보안 취약점 모니터링*

*2.1 모니터링 체계 개요*
출시 후 지속적인 사이버 보안 취약점 모니터링을 수행하여, 알려진 취약점 및 새로운 위협을 조기에 식별하고 대응할 수 있도록 한다.

*2.2 CVE 추적*
- *모니터링 소스*: NVD(National Vulnerability Database)
- *모니터링 주기*: 월 1회 정기 모니터링, Critical 보안 권고 발생 시 즉시 확인
- *모니터링 대상*:
  - 본 제품이 사용하는 JavaScript 런타임 환경(웹 브라우저 엔진) 관련 CVE
  - DICOM 파서 라이브러리 관련 CVE
  - WebGL 렌더링 관련 CVE
  - 빌드 도구(Vite/Webpack) 및 개발 의존성 관련 CVE
- *프로세스*:
  1) NVD 및 GitHub Security Advisory에서 제품 관련 CVE 검색
  2) 식별된 CVE를 SMP-03 평가 체계에 따라 위험도 평가
  3) 평가 결과를 취약점 관리 대장에 기록
  4) 필요 시 패치 계획 수립 및 이해관계자 통지

*2.3 SBOM(Software Bill of Materials) 검토*
- *SBOM 관리*:
  - 제품 릴리즈 시점의 SBOM을 문서화하여 유지 (CMP PLAYG-1234 참조)
  - SBOM에는 모든 직접 및 간접(전이적) 의존성 포함
  - 각 컴포넌트의 이름, 버전, 공급자, 라이선스 정보 포함
- *SBOM 갱신*:
  - 의존성 업데이트 시 SBOM 갱신
  - 분기 1회 SBOM 정밀 검토 수행
  - SBOM 기반으로 알려진 취약점 교차 확인

*2.4 의료기기 보안 권고 모니터링*
- *모니터링 소스*:
  - FDA Medical Device Safety Communications
  - ICS-CERT(Industrial Control Systems Cyber Emergency Response Team) 권고
  - 제조사 자체 보안 로그 및 사용자 보고
- *모니터링 주기*: 월 1회 정기 확인
- *대응 절차*:
  1) 관련 권고 식별 시 위험도 평가(SMP-03)
  2) 영향을 받는 컴포넌트 식별 (SBOM 활용)
  3) 완화 조치 수립 및 SMP-04에 따라 정보 제공

*3. [SMP-03] 취약점 평가 및 대응 체계*

*3.1 CVSS(Common Vulnerability Scoring System) 평가*
- 식별된 모든 취약점에 대해 CVSS v3.1 기반 위험도 평가를 수행한다.
- *평가 항목*:
  - Base Score: 취약점 자체의 심각도 (공격 벡터, 복잡도, 권한 필요 여부 등)
  - Temporal Score: 현재 익스플로잇 가능성 및 완화 조치 존재 여부
  - Environmental Score: Simple CBCT Viewer 환경 특성 반영 (오프라인 동작, 로컬 파일 처리)

*3.2 위험도 분류 및 대응 기준*

||CVSS 점수||등급||대응 기준||
|9.0 - 10.0|Critical|24시간 이내 대응, 즉시 패치 계획 수립|
|7.0 - 8.9|High|72시간 이내 대응, 30일 이내 패치|
|4.0 - 6.9|Medium|14일 이내 대응, 90일 이내 패치|
|0.1 - 3.9|Low|30일 이내 대응, 다음 정식 릴리즈에 반영|

*3.3 패치 주기*
- *정기 패치*: 분기 1회 정기 보안 업데이트 (의존성 업데이트 포함)
- *긴급 패치*: Critical/High 취약점 식별 시 위 대응 기준에 따른 긴급 패치
- *패치 프로세스*:
  1) 취약점 식별 및 CVSS 평가
  2) 패치 또는 완화 조치 개발 (개발 환경에서 테스트)
  3) 단위/통합 테스트 수행 (SR-1 ~ SR-12 회귀 테스트)
  4) 품질 관리자 승인 (DP-PLAYG-1231 섹션 5.2 참조)
  5) 형상 관리 프로세스에 따라 릴리즈 (PLAYG-1234 CMP 참조)
  6) SMP-04에 따라 사용자에게 정보 제공

*3.4 패치 불가 시 완화 조치*
- 패치가 즉시 불가능한 경우 다음 완화 조치를 검토한다:
  - 기능 제한 또는 비활성화
  - 사용자 가이드에 보안 주의사항 추가
  - 네트워크 격리 권고 (본 제품은 오프라인 동작이 원칙이므로 SR-7 참조)

*4. [SMP-04] 정보 제공 및 공지 절차*

*4.1 고객 알림 메커니즘*
- *알림 채널*:
  - 이메일 공지: 등록된 고객 대상 이메일 발송
  - 제품 웹사이트 공지: 보안 업데이트 게시판에 공지
  - 제품 내 알림: 업데이트 가능 시 애플리케이션 내 표시 (향후 기능)
- *알림 시기*:
  - Critical/High 취약점: 패치 배포 후 72시간 이내 알림
  - Medium/Low 취약점: 정기 보안 업데이트 시 알림
  - 패치 불가 시 완화 조치 안내: 즉시 알림

*4.2 리포팅 투명성*
- *보안 권고(Security Advisory) 작성*:
  - 취약점 설명, 영향 범위, CVSS 점수, 완화 조치, 패치 버전 명시
  - CVE 번호가 할당된 경우 이를 참조
- *정기 보안 보고*:
  - 분기 1회 보안 상태 보고서 작성
  - 보고서 내용: 모니터링 결과, 대응 이력, 미해결 취약점 현황
- *규제 기관 보고*:
  - FDA: 취약점이 환자 안전에 영향을 미치는 경우 적극적 사후 감시 보고
  - EU MDR: 심각 사건(Vigilance) 발생 시 EU Competent Authority에 보고

*4.3 취약점 공개 정책*
- *조율된 취약점 공개(Coordinated Disclosure)* 원칙 적용
- 외부 보안 연구자의 취약점 보고를 위한 채널 제공 (보안 이메일)
- 패치 배포 전 취약점 상세 공개 지연 (최대 90일)

*5. [SMP-05] 써드파티 및 오픈소스 유지보수*

*5.1 써드파티 소프트웨어 관리 체계*
- 본 제품은 JavaScript(ES6+) 기반 웹 애플리케이션으로, npm/pnpm을 통한 오픈소스 의존성 관리 (DP-PLAYG-1231 섹션 2.2 참조)
- *관리 대상*:
  - 직접 의존성(Direct Dependencies): package.json에 명시된 패키지
  - 간접 의존성(Transitive Dependencies): 직접 의존성이 참조하는 패키지
  - 개발 의존성(Dev Dependencies): 빌드 도구, 테스트 프레임워크 등

*5.2 오픈소스 라이선스 및 보안 검토*
- *도입 전 검토*:
  - 라이선스 호환성 검토 (GPL 등 강한 카피레프트 라이선스 사용 제한)
  - 알려진 보안 취약점 존재 여부 확인 (npm audit, Snyk 등)
  - 프로젝트 활성도 및 유지보수 상태 확인
- *정기 검토*:
  - 분기 1회 사용 중인 오픈소스 컴포넌트의 라이선스 및 보안 상태 재검토
  - npm audit 실행 결과 확인 및 조치

*5.3 의존성 업데이트 전략*
- *정기 업데이트*: 분기 1회 의존성 버전 업데이트 검토
- *보안 업데이트*: 취약점 발견 시 즉시 해당 의존성 업데이트
- *업데이트 절차*:
  1) 의존성 변경 시 회귀 테스트 수행 (SR-1 ~ SR-12)
  2) SBOM 갱신
  3) 품질 관리자 승인 후 릴리즈

*5.4 지원 중단(EOL) 컴포넌트 관리*
- EOL(End-of-Life) 또는 더 이상 유지보수되지 않는 오픈소스 컴포넌트 식별
- 대체 컴포넌트 평가 및 교체 계획 수립
- 교체 불가 시 추가 완화 조치 검토 (격리, 모니터링 강화 등)

*6. [SMP-06] 사고 대응 (Incident Response) 절차*

*6.1 사고 대응 체계 개요*
사이버 보안 사고 발생 시 신속하고 체계적인 대응을 위해 다음 단계별 사고 대응 절차를 수립한다. 본 절차는 ISO 14971:2019 위험 관리(RMP-PLAYG-1232) 및 IEC 62304 유지보수 프로세스와 연계하여 운영된다.

*6.2 사고 대응 단계*

*단계 1: 식별 (Identification)*
- 사이버 보안 사고 감지 소스:
  - SMP-02 모니터링 체계를 통한 취약점 식별
  - 사용자 또는 외부 보안 연구자 신고
  - 규제 기관(FDA, EU Competent Authority) 권고
- 식별 시 즉시 사고 대응 책임자(품질 관리자)에게 보고
- 사고 심각도 분류: Critical / High / Medium / Low

*단계 2: 격리 및 분석 (Containment and Analysis)*
- 영향 범위 분석:
  - 영향을 받는 소프트웨어 버전 식별
  - 영향을 받는 기능(SR-1 ~ SR-12) 평가
  - 환자 안전 영향 평가 (RMP-PLAYG-1232 참조)
- 임시 완화 조치 적용:
  - 필요 시 기능 제한 또는 비활성화
  - 오프라인 동작 환경에서의 추가 격리 조치

*단계 3: 제거 및 복구 (Eradication and Recovery)*
- 근본 원인 분석 및 취약점 수정
- 패치 개발 및 테스트 (SMP-03 패치 프로세스 준용)
- 수정된 소프트웨어 배포
- 배포 후 모니터링 강화

*단계 4: 사후 관리 (Post-Incident Activity)*
- 사고 보고서 작성 (원인, 영향, 대응 내역, 재발 방지 대책)
- 사고 대응 절차 개선점 도출
- 위험 관리 파일 갱신 (RMP-PLAYG-1232 연계)

*6.3 사고 대응 역할*

||역할||책임자||주요 책임||
|사고 대응 책임자|품질 관리자|사고 심각도 분류, 대응 총괄, 규제 기관 보고|
|개발팀|소프트웨어 개발자|패치 개발, 테스트, 배포|
|위험 관리 담당자|위험 관리자|환자 안전 영향 평가, 위험 관리 파일 갱신|

*6.4 규제 기관 보고*
- FDA: 환자 안전에 직간접적 영향이 있는 사고 시 즉시 보고
- EU MDR: 심각 사건 발생 시 EU Competent Authority 및 인증 기관(Notified Body)에 보고
- 보고 내용: 사고 설명, 영향 받는 기기, 취한 조치, 예방 대책
