*1. [DP-01] 개발 프로세스 개요*

본 프로젝트는 IEC 62304:2006+AMD1:2015 표준을 준수하여 소프트웨어 개발 수명 주기를 관리한다.
Simple CBCT Viewer는 IEC 62304 기준 Class A(안전 등급) 소프트웨어로 분류되므로(PLAYG-1230 Classification 근거),
표준에서 요구하는 Class A 필수 프로세스 활동을 적용한다.

*1.1 수명 주기 모델*

- V-Model 기반 개발 프로세스를 채택한다.
  - 좌측(설계 단계): 요구사항 분석 -> 아키텍처 설계 -> 상세 설계 -> 구현
  - 우측(검증 단계): 단위 테스트 -> 통합 테스트 -> 시스템 테스트 -> 사용자 인수 테스트
  - 각 설계 단계와 대응되는 테스트 단계 간 추적성(Traceability)을 유지한다.
- Agile 실무 요소를 보완적으로 적용한다.
  - 반복적(Iterative) 개발 방식으로 기능 단위 증분(Increment)을 관리한다.
  - 각 Sprint(2주 단위)마다 요구사항 점검 및 진척 상황을 공유한다.

*1.2 IEC 62304 Class A 필수 활동 매핑*

||IEC 62304 절차||해당 활동||산출물||
|제5.1 소프트웨어 개발 계획|본 문서(SDP)|SDP 문서|
|제5.2 소프트웨어 요구사항 분석|SyRS 기반 SRS 작성|SRS 문서|
|제5.3 소프트웨어 아키텍처 설계|아키텍처 설계 문서 작성|SAD 문서|
|제5.5 소프트웨어 구현|코딩 및 단위 구현|소스코드|
|제5.6 소프트웨어 검증|단위/통합/시스템 테스트|테스트 보고서|
|제5.7 소프트웨어 검증(Validation)|사용자 요구사항 대비 검증|검증 보고서|
|제5.8 소프트웨어 배포|릴리즈 패키징 및 배포|릴리즈 노트|
|제6 소프트웨어 유지보수 프로세스|유지보수 계획 수립|유지보수 계획서|

*1.3 마일스톤*

- MS1: 요구사항 분석 완료 (SyRS 확정)
- MS2: 아키텍처 설계 완료 (SAD 확정)
- MS3: 구현 완료 (모든 기능 구현 및 단위 테스트 통과)
- MS4: 시스템 테스트 완료 (통합/시스템 테스트 통과)
- MS5: 검증(Validation) 완료 및 배포 (사용자 인수 테스트 통과)

주요 시스템 요구사항(12건)은 SyRS(SR-1 ~ SR-12)에 정의되어 있으며, 각 마일스톤에서 추적성을 보장한다.

*2. [DP-02] 개발 환경 및 도구*

본 프로젝트의 개발 환경은 Simple CBCT Viewer의 기술적 특성(웹 브라우저 기반, 오프라인 동작,
DICOM 3.0 파싱, WebGL 렌더링)에 맞게 구성한다.

*2.1 프로그래밍 언어 및 프레임워크*

- 프론트엔드 언어: JavaScript (ES6+)
  - 근거: IU-08 작동 원리에 명시된 JavaScript 기반 DICOM 파서 및 HTML5 Canvas/WebGL 렌더링
- 3D 렌더링: WebGL (SR-11: WebGL 기반 3D 볼륨 렌더링 요구사항)
- DICOM 파싱: JavaScript 기반 DICOM 3.0 파서 (SR-1 근거)
- 프레임워크: 별도의 서버 인프라가 불필요한 정적 웹 애플리케이션 구조 (IU-07 오프라인 동작 근거)

*2.2 개발 도구*

||구분||도구||용도||
|IDE / 편집기|VS Code (Visual Studio Code)|소스코드 편집|
|버전 관리|Git|소스코드 형상 관리, 브랜치 관리|
|원격 저장소|GitHub / GitLab|협업용 원격 저장소, 코드 리뷰, CI/CD 연동|
|빌드 도구|Vite 또는 Webpack|프론트엔드 번들링, 개발 서버, 프로덕션 빌드|

*2.3 대상 실행 환경*

- 지원 브라우저: 최신 버전 Chrome, Edge, Firefox (SR-8, IU-07 근거)
- 런타임 요구사항: 인터넷 연결 불필요 (SR-7 오프라인 동작)
- 최소 하드웨어: RAM 8GB 이상 권장 (IU-12 주의사항 근거)

*3. [DP-03] 코딩 및 설계 표준*

*3.1 코딩 스타일 가이드*

- JavaScript: Airbnb JavaScript Style Guide 또는 Google JavaScript Style Guide를 기본으로 채택
  - 들여쓰기: 스페이스 2칸
  - 명명 규칙: camelCase (변수/함수), PascalCase (클래스)
  - 세미콜론: 필수 사용
  - 따옴표: 싱글 쿼트 기본
- HTML/CSS: W3C 표준 준수
- ESLint 및 Prettier를 통한 코드 스타일 자동화 (DP-02 참조)

*3.2 아키텍처 원칙*

- 모듈식 구조: 기능별 독립 모듈 분리 (DICOM 파서, 렌더링 엔진, UI 컴포넌트, 측정 도구)
- 관심사 분리(Separation of Concerns): 데이터 처리 로직과 UI 렌더링 로직의 명확한 분리
- 의존성 주입: 외부 라이브러리 의존성은 명시적으로 관리하고 최소화
- 재사용성: 공통 유틸리티 함수 및 컴포넌트의 재사용성 극대화
- 오프라인 독립성: 모든 핵심 기능이 네트워크 연결 없이 동작하도록 설계 (SR-7 근거)

*3.3 코드 리뷰 규칙*

- 모든 코드 변경은 Pull Request(PR) 기반으로 관리
- 최소 1인 이상의 리뷰어 승인 후 병합 (Class A 요구사항)
- 리뷰 체크리스트: 기능 정확성, 코딩 표준 준수, 보안 취약점, 테스트 커버리지

*3.4 문서화 규칙*

- 함수/모듈 단위 JSDoc 주석 필수 작성
- 공개 API 및 주요 인터페이스에 대한 문서화
- 아키텍처 다이어그램 및 모듈 간 의존성 문서 유지

*4. [DP-04] 소프트웨어 위험 관리 계획 연계 방안*

본 프로젝트는 IEC 62304 제7절(소프트웨어 위험 관리) 및 ISO 14971(의료기기 위험 관리) 요구사항에 따라
소프트웨어 위험 관리 활동을 수행한다.

*4.1 위험 관리 프로세스 연계*

- 소프트웨어 위험 분석은 개발 수명 주기의 각 단계와 연계하여 수행
  - 요구사항 단계: SyRS 각 항목별 잠재 위험 식별
  - 설계 단계: 아키텍처 및 상세 설계에서의 위험 완화 조치 반영
  - 구현 단계: 코딩 시 위험 완화 조치 구현
  - 테스트 단계: 식별된 위험 항목에 대한 검증 테스트 수행

*4.2 주요 위험 영역 (SyRS 기반)*

- 영상 렌더링 왜곡: DICOM 파싱 또는 렌더링 오류로 인한 영상 왜곡 가능성 (SR-1, SR-2)
- 거리 측정 오류: 픽셀 간격(Pixel Spacing) 해석 오류로 인한 측정 부정확 (SR-6)
- 환자 정보 유출: DICOM 파일 내 PHI 데이터의 외부 전송 가능성 (SR-10)
- 경고 문구 누락: 사용자 제약 경고가 표시되지 않을 가능성 (SR-9)
- 브라우저 호환성 문제: 특정 브라우저에서 기능 동작 불량 (SR-8)

*4.3 Class A 위험 관리 요구사항*

- 본 소프트웨어는 Class A로 분류되므로 IEC 62304 제7절의 Class A 필수 활동을 적용
  - 소프트웨어로 인한 위험 원인 분석
  - 위험 완화 조치 문서화
  - 위험 관리 결과 추적성 유지
- 위험 관리 결과는 별도의 위험 관리 보고서(Risk Management Report)에 기록

*4.4 위험 완화 전략*

- 입력 데이터 검증: DICOM 파일 유효성 검사 및 예외 처리
- 메타데이터 검증: Pixel Spacing 등 측정 관련 메타데이터의 유효성 확인
- 외부 전송 차단: 네트워크 요청 발생 시 코드 레벨 차단 및 정적 분석으로 검증
- UI 경고 표시: 애플리케이션 로드 시 진단 불가 경고 문구 의무 표시
- 크로스 브라우저 테스트: 지원 브라우저(Chrome, Edge, Firefox) 대상 E2E 테스트 수행

*6. [DP-06] 형상 관리 및 빌드 배포 계획 요약*

*6.1 형상 관리 (Configuration Management)*

- 형상 관리 도구: Git (DP-02 참조)
- 브랜치 전략:
  - main: 프로덕션 릴리즈 브랜치
  - develop: 개발 통합 브랜치
  - feature/*: 기능 개발 브랜치
  - release/*: 릴리즈 준비 브랜치
  - hotfix/*: 긴급 수정 브랜치

*6.2 형상 항목 (Configuration Items)*

- 소스코드 (모든 모듈)
- 빌드 스크립트 및 구성 파일 (package.json, webpack/vite 설정 등)
- 테스트 코드 및 테스트 데이터
- 개발 산출물 문서 (SRS, SAD, 테스트 보고서 등)
- 서드파티 라이브러리 목록 및 버전 정보

*6.3 버전 관리 정책*

- 의미적 버전 관리(Semantic Versioning) 적용: MAJOR.MINOR.PATCH
  - MAJOR: 호환성이 깨지는 변경
  - MINOR: 기능 추가 (하위 호환 유지)
  - PATCH: 버그 수정 (하위 호환 유지)
- 각 릴리즈마다 고유 버전 태그(Tag) 부여

*6.4 빌드 및 배포 프로세스*

- 빌드 파이프라인:
  - 코드 체크아웃 -> 의존성 설치 -> 정적 분석(ESLint) -> 단위 테스트 -> 프로덕션 빌드 -> 아티팩트 생성
- 배포 산출물:
  - 정적 웹 리소스 (HTML, CSS, JavaScript 번들)
  - 단일 배포 패키지 (ZIP 또는 압축 아카이브)
  - 릴리즈 노트 및 검증 기록
- 릴리즈 절차:
  - release 브랜치 생성 -> 통합 테스트 -> 품질 관리자 승인 -> main 브랜치 병합 -> 버전 태그 부여 -> 배포

*6.5 CI/CD (지속적 통합/지속적 배포)*

- CI: Pull Request 시 자동으로 정적 분석 및 단위 테스트 실행
- CD: main 브랜치 병합 시 자동 빌드 및 아티팩트 생성
- CI/CD 도구: GitHub Actions 또는 GitLab CI

*6.6 백업 및 복구*

- 원격 저장소(GitHub/GitLab)를 통한 소스코드 백업 보장
- 산출물 문서는 형상 관리 저장소 내 docs/ 디렉토리에 관리
- 정기적 백업 검증 수행
