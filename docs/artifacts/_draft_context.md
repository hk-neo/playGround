## [IU-01] 제품명 (Product Name)

**Simple CBCT Viewer**

본 제품은 웹 브라우저 기반의 CBCT(Cone Beam Computed Tomography) 영상 뷰어로, 사용자가 로컬 환경에서 DICOM 파일을 직접 불러와 2D/3D 영상을 확인할 수 있는 간소화된 의료 영상 열람 소프트웨어입니다.

---

## [IU-02] 의도된 목적 (Intended Purpose)

Simple CBCT Viewer는 CBCT 장비에서 획득한 DICOM 형식의 의료 영상을 웹 브라우저 상에서 열람 및 분석할 수 있도록 지원하는 독립형 영상 뷰어입니다. 본 제품은 진단 목적이 아닌, 기존에 획득된 CBCT 영상의 시각적 확인 및 영상 품질 검토를 위한 보조 도구로 설계되었습니다.

사용자는 로컬 파일 시스템에서 DICOM 파일 또는 폴더를 직접 선택하여 불러올 수 있으며, 별도의 서버 인프라나 네트워크 연결 없이도 영상 열람이 가능합니다.

---

## [IU-03] 적응증 (Indication for Use)

본 제품은 CBCT 영상을 활용하는 치과영역(구강악안면방사선)의 영상 확인에 사용됩니다. 구체적으로 다음과 같은 용도로 활용됩니다:

- 치과용 CBCT 영상의 2D 단면(Axial, Sagittal, Coronal) 및 MPR(Multi-Planar Reconstruction) 열람
- 치근단, 매복치, 치조골 상태 등 치과 임상 영상의 확인
- 기존에 획득된 DICOM 영상의 시각적 검토 및 품질 확인

**주의**: 본 제품은 단독 진단 도구가 아니며, 영상의 최종 임상 해석은 반드시 전문의에 의해 수행되어야 합니다.

---

## [IU-04] 대상 환자군 (Intended Patient Population)

본 제품은 환자에게 직접 사용되는 장비가 아니며, 이미 획득된 환자의 CBCT 영상 데이터를 열람하는 소프트웨어입니다. 따라서 특정 환자군에 대한 제한은 없으며, CBCT 촬영이 수행된 모든 연령대의 환자 영상을 대상으로 합니다.

- 연령: 전 연령 (소아, 성인, 노인)
- 성별: 제한 없음
- 특이사항: 치과용 CBCT 촬영이 수행된 환자의 영상 데이터

---

**해당 없음 (Not Applicable)**

## [IU-08] 작동 원리 (Principle of Operation)

Simple CBCT Viewer는 다음과 같은 원리로 동작합니다:

1. **DICOM 파일 로딩**: 사용자가 로컬 파일 시스템에서 DICOM 파일(.dcm) 또는 DICOM 파일이 포함된 폴더를 선택하여 웹 브라우저로 불러옵니다.

2. **영상 데이터 파싱**: 불러온 DICOM 파일의 헤더 메타데이터와 픽셀 데이터를 JavaScript 기반 DICOM 파서를 통해 해석합니다.

3. **영상 렌더링**: 파싱된 복셀 데이터를 활용하여 HTML5 Canvas/WebGL을 통해 2D 단면 영상(Axial, Sagittal, Coronal)을 렌더링합니다.

4. **MPR 재구성**: 3차원 볼륨 데이터로부터 임의의 단면을 실시간으로 재구성하여 다양한 방향의 영상을 제공합니다.

5. **사용자 인터랙션**: 마우스 및 키보드 입력을 통해 영상의 스크롤, 확대/축소, 윈도우 레벨 조절, 측정 등의 기능을 제공합니다.

## [IU-09] 금기 사항 (Contraindications)

- **진단 목적의 단독 사용**: 본 제품은 단독 진단 도구로 사용해서는 안 됩니다. 최종 임상 진단은 반드시 유자격 전문의의 판독을 통해 이루어져야 합니다.
- **응급 상황에서의 의사결정**: 응급 상황에서 본 제품의 영상만을 근거로 임상적 의사결정을 내려서는 안 됩니다.
- **비의료 목적의 오용**: 본 제품을 의료 영상 진단 외의 목적으로 오용해서는 안 됩니다.
- **비전문가의 임상 해석**: 의료 면허가 없는 비전문가가 영상을 임상적으로 해석하여 진단이나 치료 방침을 결정해서는 안 됩니다.
- **영상 획득 대체**: 본 제품은 CBCT 영상 획득 장비를 대체할 수 없습니다.

## [IU-11] 임상적 이점 (Clinical Benefit)

- **접근성 향상**: 웹 브라우저만으로 CBCT 영상을 열람할 수 있어, 별도의 전용 소프트웨어 설치 없이 즉시 사용 가능합니다.
- **오프라인 사용**: 네트워크 연결 없이도 로컬 DICOM 파일을 불러와 영상을 확인할 수 있어, 인프라가 제한된 환경에서도 활용 가능합니다.
- **직관적 인터페이스**: 간소화된 UI를 통해 전문 교육 없이도 기본적인 영상 열람이 가능합니다.
- **다단면 동시 열람**: Axial, Sagittal, Coronal 단면을 동시에 확인하여 영상의 공간적 이해도를 향상시킵니다.
- **영상 품질 사전 검토**: CBCT 촬영 후 즉시 영상 품질을 확인하여 재촬영 필요 여부를 신속히 판단할 수 있습니다.
- **환자 설명 보조**: 환자에게 치료 계획을 설명할 때 시각적 자료로 활용 가능합니다.

## [IU-12] 경고 및 주의사항 (Warnings and Precautions)

### 경고 (Warnings)

1. 본 제품은 단독 진단 도구가 아닙니다. 영상의 임상적 해석 및 최종 진단은 반드시 유자격 전문의가 수행해야 합니다.
2. 본 제품에서 표시되는 영상의 색상, 명암, 측정값은 참고용이며, 절대적 임상 수치로 간주해서는 안 됩니다.
3. 모니터의 색상 보정이 해제된 경우 영상의 정확도가 저하될 수 있습니다. 의료용 모니터 사용을 권장합니다.

### 주의사항 (Precautions)

1. 대용량 DICOM 데이터를 로딩할 때 브라우저의 메모리 부족으로 인해 성능이 저하될 수 있습니다. 충분한 메모리(RAM 8GB 이상) 권장.
2. 지원되지 않는 DICOM 형식이나 손상된 파일은 정상적으로 로딩되지 않을 수 있습니다.
3. 웹 브라우저의 최신 버전을 사용하십시오. 구형 브라우저에서는 일부 기능이 동작하지 않을 수 있습니다.
4. 환자 식별 정보(PHI)가 포함된 DICOM 파일을 열람 시 개인정보 보호 규정을 준수하십시오.
5. 본 제품의 영상 품질은 원본 CBCT 장비의 획득 설정에 따라 달라질 수 있습니다.
6. 네트워크를 통한 전송 기능이 없으므로, 영상 공유가 필요한 경우 별도의 수단을 이용하십시오.

*4. [DP-04] 소프트웨어 위험 관리 계획 연계 방안*

본 프로젝트는 IEC 62304 제7절(소프트웨어 위험 관리) 및 ISO 14971(의료기기 위험 관리) 요구사항에 따라
소프트웨어 위험 관리 활동을 수행한다.

*4.1 위험 관리 프로세스 연계*

- 소프트웨어 위험 분석은 개발 수명 주기의 각 단계와 연계하여 수행
  - 요구사항 단계: SyRS 각 항목별 잠재 위험 식별
  - 설계 단계: 아키텍처 및 상세 설계에서의 위험 완화 조치 반영
  - 구현 단계: 코딩 시 위험 완화 조치 구현
  - 테스트 단계: 식별된 위험 항목에 대한 검증 테스트 수행

*4.2 주요 위험 영역 (SyRS 기반)*

- 영상 렌더링 왜곡: DICOM 파싱 또는 렌더링 오류로 인한 영상 왜곡 가능성 (SR-1, SR-2)
- 거리 측정 오류: 픽셀 간격(Pixel Spacing) 해석 오류로 인한 측정 부정확 (SR-6)
- 환자 정보 유출: DICOM 파일 내 PHI 데이터의 외부 전송 가능성 (SR-10)
- 경고 문구 누락: 사용자 제약 경고가 표시되지 않을 가능성 (SR-9)
- 브라우저 호환성 문제: 특정 브라우저에서 기능 동작 불량 (SR-8)

*4.3 Class A 위험 관리 요구사항*

- 본 소프트웨어는 Class A로 분류되므로 IEC 62304 제7절의 Class A 필수 활동을 적용
  - 소프트웨어로 인한 위험 원인 분석
  - 위험 완화 조치 문서화
  - 위험 관리 결과 추적성 유지
- 위험 관리 결과는 별도의 위험 관리 보고서(Risk Management Report)에 기록

*4.4 위험 완화 전략*

- 입력 데이터 검증: DICOM 파일 유효성 검사 및 예외 처리
- 메타데이터 검증: Pixel Spacing 등 측정 관련 메타데이터의 유효성 확인
- 외부 전송 차단: 네트워크 요청 발생 시 코드 레벨 차단 및 정적 분석으로 검증
- UI 경고 표시: 애플리케이션 로드 시 진단 불가 경고 문구 의무 표시
- 크로스 브라우저 테스트: 지원 브라우저(Chrome, Edge, Firefox) 대상 E2E 테스트 수행

*2.1 위험 식별 (Risk Identification)*
- SyRS 각 항목(SR-1 ~ SR-12) 및 Intended Use(PLAYG-1237)를 기반으로 잠재적 위험원(Hazard)을 식별
- 위험원 식별 기법: FMEA(실패 유형 및 영향 분석)
- 식별 대상: 정상 사용 조건 및 합리적으로 예측 가능한 오사용
- 주요 위험 영역 (DP-PLAYG-1231 섹션 4.2 참조):
  - 영상 렌더링 왜곡: DICOM 파싱 또는 렌더링 오류로 인한 영상 왜곡 가능성 (SR-1, SR-2)
  - 거리 측정 오류: Pixel Spacing 해석 오류로 인한 측정 부정확 (SR-6)
  - 환자 정보 유출: DICOM 파일 내 PHI 데이터의 외부 전송 가능성 (SR-10)
  - 경고 문구 누락: 사용자 제약 경고가 표시되지 않을 가능성 (SR-9)
  - 브라우저 호환성 문제: 특정 브라우저에서 기능 동작 불량 (SR-8)

*2.2 위험 분석 및 평가 (Risk Analysis and Evaluation)*
- 식별된 각 위험원에 대해 위해 사건(Hazardous Situation)을 정의
- 각 위해 사건에 대해 Severity(심각도)와 Probability(발생 확률)를 평가
- Risk = Severity x Probability 매트릭스를 사용하여 위험 수준 산출
- 산출된 위험 수준을 RMP-03의 수용 기준과 비교하여 수용 가능성 판단

*2.3 위험 통제 (Risk Control)*
- 수용 불가능한 위험에 대해 위험 통제 조치를 수립
- 통제 조치 우선순위 (ISO 14971 제거 순서):
  1) 본질적 안전 설계 (Inherent Safety by Design)
  2) 보호 조치 (Protective Measures)
  3) 정보 전달 (Information for Safety)
- 통제 조치 구현 후 잔여 위험을 재평가

*2.4 잔여 위험 검토 (Residual Risk Review)*
- 모든 위험 통제 조치 적용 후 잔여 위험을 종합적으로 검토
- 개별 잔여 위험이 수용 기준을 충족하는지 확인
- 전체 잔여 위험의 누적 효과를 평가하여 전반적 잔여 위험 수용 여부 판단
- 위험-편익 분석(Risk-Benefit Analysis) 수행 (필요 시)

*2.5 프로세스 흐름*
위험 식별 -> 위해 사건 정의 -> 위험 분석(Severity/Probability 평가) -> 위험 평가(수용 기준 비교) -> 위험 통제 조치 수립 -> 통제 조치 구현 -> 잔여 위험 재평가 -> 전반적 잔여 위험 검토 -> 위험 관리 보고서 작성

*3.4 위험 수준별 조치*
- *수용 (Acceptable)*: 추가 조치 불필요. 모니터링만 수행.
- *낮음 (Low)*: ALARP 원칙 적용. 합리적으로 실현 가능한 통제 조치 구현.
- *중간 (Medium)*: 위험 통제 조치 필수. 통제 후 잔여 위험 재평가.
- *높음 (High)*: 즉각적 위험 통제 조치 필수. 위험-편익 분석 필요. 통제 후에도 높음 시 설계 변경.

*3.5 Class A 소프트웨어 위험 수용 기준*
본 소프트웨어는 IEC 62304 Class A로 분류(PLAYG-1290)되므로, 소프트웨어로 인한 위험이 낮음(Low) 이하인 경우 수용 가능하다. 중간(Medium) 이상의 위험은 반드시 통제 조치를 구현해야 하며, ALARP 원칙에 따라 합리적으로 실현 가능한 모든 통제 조치를 적용한다.

*5. [RMP-05] 위험 관리 활동 계획*

*5.1 초기 위험 분석 (요구사항 단계)*
- *시기*: SyRS 작성 완료 후 (MS1 마일스톤)
- *활동*:
  - SyRS(SR-1 ~ SR-12) 각 항목별 잠재적 위험원 식별
  - Intended Use(PLAYG-1237) 기반 위험원 식별
  - 초기 Severity/Probability 평가 및 위험 매트릭스 적용
  - 위험 통제 요구사항을 설계 입력으로 전환
- *산출물*: 초기 위험 분석 보고서

*5.2 설계/구현 단계 지속 위험 관리*
- *시기*: 아키텍처 설계(MS2) ~ 구현 완료(MS3)
- *활동*:
  - 아키텍처 설계 검토 시 위험 완화 조치 반영 여부 확인
  - 상세 설계 단계에서 위험 통제 조치의 구현 방법 정의
  - 구현 단계에서 코드 레벨 위험 통제 조치 적용:
    - 입력 데이터 검증: DICOM 파일 유효성 검사 및 예외 처리 (SR-1, SR-2)
    - 메타데이터 검증: Pixel Spacing 등 측정 메타데이터 유효성 확인 (SR-6)
    - 외부 전송 차단: 네트워크 요청 코드 레벨 차단 및 정적 분석 검증 (SR-10)
    - UI 경고 표시: 진단 불가 경고 문구 의무 표시 (SR-9)
  - 위험 통제 조치별 단위 테스트 작성 및 실행
- *산출물*: 갱신된 위험 등록부

*5.3 검증 단계 위험 관리*
- *시기*: 단위/통합/시스템 테스트 (MS4)
- *활동*:
  - 식별된 위험 항목에 대한 검증 테스트 수행
  - 위험 통제 조치의 실효성 검증:
    - 크로스 브라우저 테스트: Chrome, Edge, Firefox (SR-8)
    - 측정 정확도 테스트: 거리 측정 오차 범위 확인 (SR-6)
    - 오프라인 동작 테스트: 네트워크 차단 상태에서 기능 확인 (SR-7)
    - 보안 테스트: 외부 데이터 전송 여부 정적 분석 (SR-10)
  - 검증 결과 기반 잔여 위험 재평가
- *산출물*: 검증 테스트 결과, 갱신된 위험 등록부

*5.4 출시 후 모니터링 (Post-Market Monitoring)*
- *시기*: 제품 출시 후 지속
- *활동*:
  - 사용자 피드백 및 불만 사항 수집 체계 운영
  - 식별된 새로운 위험원에 대한 위험 분석 수행
  - 기존 위험 평가의 적절성 정기 재검토
  - 필요 시 위험 관리 계획 업데이트 및 추가 통제 조치 구현
  - 생산 후 모니터링 데이터를 다음 버전 위험 분석에 반영
- *산출물*: 생산 후 모니터링 보고서, 위험 관리 계획 개정판 (필요 시)

  - 소프트웨어 설계 요소 (SAD 모듈)

*6.4 문서 갱신 주기*

- 위험 관리 보고서(RMR): 모든 위험 관리 활동 완료 후, 제품 출시 전 최종 작성
- FMEA 분석서: 요구사항 변경 또는 설계 변경 시 갱신

*6.5 승인 및 보관*

- 이전 버전 문서는 형상 관리 이력을 통해 추적 가능

*3. [SMP-03] 취약점 평가 및 대응 체계*

*3.1 CVSS(Common Vulnerability Scoring System) 평가*
- 식별된 모든 취약점에 대해 CVSS v3.1 기반 위험도 평가를 수행한다.
- *평가 항목*:
  - Base Score: 취약점 자체의 심각도 (공격 벡터, 복잡도, 권한 필요 여부 등)
  - Temporal Score: 현재 익스플로잇 가능성 및 완화 조치 존재 여부
  - Environmental Score: Simple CBCT Viewer 환경 특성 반영 (오프라인 동작, 로컬 파일 처리)

*3.2 위험도 분류 및 대응 기준*

||CVSS 점수||등급||대응 기준||
|9.0 - 10.0|Critical|24시간 이내 대응, 즉시 패치 계획 수립|
|7.0 - 8.9|High|72시간 이내 대응, 30일 이내 패치|
|4.0 - 6.9|Medium|14일 이내 대응, 90일 이내 패치|
|0.1 - 3.9|Low|30일 이내 대응, 다음 정식 릴리즈에 반영|

*3.3 패치 주기*
- *정기 패치*: 분기 1회 정기 보안 업데이트 (의존성 업데이트 포함)
- *긴급 패치*: Critical/High 취약점 식별 시 위 대응 기준에 따른 긴급 패치
- *패치 프로세스*:
  1) 취약점 식별 및 CVSS 평가
  2) 패치 또는 완화 조치 개발 (개발 환경에서 테스트)
  3) 단위/통합 테스트 수행 (SR-1 ~ SR-12 회귀 테스트)
  4) 품질 관리자 승인 (DP-PLAYG-1231 섹션 5.2 참조)
  5) 형상 관리 프로세스에 따라 릴리즈 (PLAYG-1234 CMP 참조)
  6) SMP-04에 따라 사용자에게 정보 제공

*3.4 패치 불가 시 완화 조치*
- 패치가 즉시 불가능한 경우 다음 완화 조치를 검토한다:
  - 기능 제한 또는 비활성화
  - 사용자 가이드에 보안 주의사항 추가
  - 네트워크 격리 권고 (본 제품은 오프라인 동작이 원칙이므로 SR-7 참조)

*6. [SMP-06] 사고 대응 (Incident Response) 절차*

*6.1 사고 대응 체계 개요*
사이버 보안 사고 발생 시 신속하고 체계적인 대응을 위해 다음 단계별 사고 대응 절차를 수립한다. 본 절차는 ISO 14971:2019 위험 관리(RMP-PLAYG-1232) 및 IEC 62304 유지보수 프로세스와 연계하여 운영된다.

*6.2 사고 대응 단계*

*단계 1: 식별 (Identification)*
- 사이버 보안 사고 감지 소스:
  - SMP-02 모니터링 체계를 통한 취약점 식별
  - 사용자 또는 외부 보안 연구자 신고
  - 규제 기관(FDA, EU Competent Authority) 권고
- 식별 시 즉시 사고 대응 책임자(품질 관리자)에게 보고
- 사고 심각도 분류: Critical / High / Medium / Low

*단계 2: 격리 및 분석 (Containment and Analysis)*
- 영향 범위 분석:
  - 영향을 받는 소프트웨어 버전 식별
  - 영향을 받는 기능(SR-1 ~ SR-12) 평가
  - 환자 안전 영향 평가 (RMP-PLAYG-1232 참조)
- 임시 완화 조치 적용:
  - 필요 시 기능 제한 또는 비활성화
  - 오프라인 동작 환경에서의 추가 격리 조치

*단계 3: 제거 및 복구 (Eradication and Recovery)*
- 근본 원인 분석 및 취약점 수정
- 패치 개발 및 테스트 (SMP-03 패치 프로세스 준용)
- 수정된 소프트웨어 배포
- 배포 후 모니터링 강화

*단계 4: 사후 관리 (Post-Incident Activity)*
- 사고 보고서 작성 (원인, 영향, 대응 내역, 재발 방지 대책)
- 사고 대응 절차 개선점 도출
- 위험 관리 파일 갱신 (RMP-PLAYG-1232 연계)

*6.3 사고 대응 역할*

||역할||책임자||주요 책임||
|사고 대응 책임자|품질 관리자|사고 심각도 분류, 대응 총괄, 규제 기관 보고|
|개발팀|소프트웨어 개발자|패치 개발, 테스트, 배포|
|위험 관리 담당자|위험 관리자|환자 안전 영향 평가, 위험 관리 파일 갱신|

*6.4 규제 기관 보고*
- FDA: 환자 안전에 직간접적 영향이 있는 사고 시 즉시 보고
- EU MDR: 심각 사건 발생 시 EU Competent Authority 및 인증 기관(Notified Body)에 보고
- 보고 내용: 사고 설명, 영향 받는 기기, 취한 조치, 예방 대책

- *태그 명명 규칙*: v[MAJOR].[MINOR].[PATCH] (예: v1.0.0)

*2.3 버전 넘버링 규칙*
의미적 버전 관리(Semantic Versioning) 적용 (DP-PLAYG-1231 섹션 6.3 참조)
- 형식: MAJOR.MINOR.PATCH
  - MAJOR: 하위 호환성이 깨지는 변경 (예: API 변경, 아키텍처 재설계)
  - MINOR: 기능 추가 (하위 호환 유지) (예: 새로운 측정 도구 추가)
  - PATCH: 버그 수정 (하위 호환 유지) (예: DICOM 파싱 오류 수정)
- 사전 릴리즈 식별자(필요 시): v1.0.0-alpha.1, v1.0.0-beta.2
- 빌드 메타데이터(필요 시): v1.0.0+build.123
- 각 릴리즈마다 고유 버전 태그(Tag)를 부여하여 형상 추적성을 보장한다.

*3. [CMP-03] 형상 통제 절차*

*3.1 브랜치 전략*
Git Flow 기반 브랜치 전략을 채택한다 (DP-PLAYG-1231 섹션 6.1 참조).
- *main*: 프로덕션 릴리즈 브랜치. 모든 릴리즈는 main 브랜치에서 수행. 병합은 release 또는 hotfix 브랜치를 통해서만 허용
- *develop*: 개발 통합 브랜치. 기능 개발 완료 후 PR을 통해 develop에 병합. CI 파이프라인이 자동 실행됨
- *feature/*: 기능 개발 브랜치. develop에서 분기 후 완료 시 PR로 develop에 병합
  - 명명 규칙: feature/[JIRA-KEY]-[설명]
- *release/*: 릴리즈 준비 브랜치. develop에서 분기 후 버전 번호 업데이트 및 최종 테스트 수행
- *hotfix/*: 긴급 수정 브랜치. main에서 분기 후 수정 완료 시 main과 develop에 병합

*3.2 코드 리뷰 절차*
- 모든 코드 변경은 Pull Request(PR) 기반으로 관리한다.
- PR 생성 시:
  1) 작성자가 변경 내용 설명, 관련 Jira 티켓 번호, 테스트 결과를 PR 설명에 기록
  2) 자동 CI 검사 실행 (정적 분석 ESLint, 단위 테스트)
  3) 최소 1인 이상의 리뷰어 지정 (IEC 62304 Class A 요구사항)
- 리뷰 체크리스트 (DP-PLAYG-1231 섹션 3.3 참조):
  - 기능 정확성: 요구사항(SR-1 ~ SR-14) 대비 구현 일치 여부
  - 코딩 표준 준수: Airbnb/Google JavaScript Style Guide 준수 여부
  - 보안 취약점: PHI 데이터 외부 전송 여부 (SR-11 환자 정보 보호)
  - 테스트 커버리지: 변경 사항에 대한 단위 테스트 포함 여부
  - 문서 업데이트: 관련 문서 갱신 필요 여부
- 승인 조건: 모든 CI 검사 통과 + 최소 1인 리뷰어 승인(Approve)
- 병합 방식: Squash and Merge 원칙 (커밋 히스토리 정리)

*3.3 변경 승인 절차*
소프트웨어 형상 항목의 변경은 다음 승인 절차를 따른다.

||변경 유형||승인 권한||비고||
|일반 기능 개발 (feature)|코드 리뷰어 승인|PR 기반 승인|
|릴리즈 병합 (release -> main)|프로젝트 관리자 + 품질 관리자 공동 승인|DP-PLAYG-1231 섹션 5.2 참조|

  2) 수정 구현 및 테스트
  3) 프로젝트 관리자 승인 (사후 리뷰 필수)
  4) main에 병합 및 버전 태그 부여
  5) develop에 역병합(cherry-pick 또는 merge)
  6) 위험 관리 담당자 통지 (RMP-PLAYG-1232 연계)

*3. [SRS-03] 전체 설명*

*3.1 제품 관점*
Simple CBCT Viewer는 치과용 CBCT 영상을 웹 브라우저에서 열람할 수 있는 독립형 소프트웨어이다.
서버 인프라가 불필요하며, 모든 데이터는 사용자의 로컬 환경에서 처리된다. (PLAYG-1237 IU-07, IU-08 참조)

*3.2 사용자 클래스*
- *주 사용자*: 치과 의사
- *부 사용자*: 치과 기공사, 치과 교육생 (교육/연구 환경)
- *공통 제약*: 본 제품은 진단 목적이 아닌 영상 열람 보조 도구임 (PLAYG-1237 IU-09 금기 사항)

*3.3 운영 환경*
- *플랫폼*: 웹 브라우저 (Chrome, Edge, Firefox 최신 버전)
- *네트워크*: 인터넷 연결 불필요 (오프라인 동작)
- *하드웨어*: RAM 8GB 이상 권장
- *입력*: 로컬 DICOM 3.0 표준 CBCT 영상 파일

*3.4 제약 조건*
- 서버 인프라 불필요 (정적 웹 애플리케이션)
- 환자 데이터의 외부 전송 금지 (PLAYG-1237 IU-12 경고)
- 진단 목적 사용 금지 (PLAYG-1237 IU-09 금기 사항)
- IEC 62304 Class A 준수 (PLAYG-1290)

*[SAD-01] 아키텍처 개요 및 설계 원칙*

*1.1 시스템 전체 아키텍처 개요*

Simple CBCT Viewer는 웹 브라우저에서 동작하는 독립형 치과용 CBCT 영상 뷰어로, 모든 데이터 처리를 클라이언트 로컬 환경에서 수행하는 단일 페이지 애플리케이션(SPA) 구조를 채택한다. 본 시스템은 Layered Architecture 패턴을 기반으로 하여 데이터 파싱 계층, 비즈니스 로직 계층, 렌더링 계층, 프레젠테이션 계층으로 분리되며, IEC 62304 Class A 요구사항과 오프라인 동작 제약조건을 충족하도록 설계되었다.

*1.2 핵심 설계 원칙*

- *단일 책임 원칙 (SRP)*: 각 모듈은 하나의 명확한 책임을 가진다 (예: DICOMParser는 파일 파싱만, Renderer는 영상 렌더링만 담당)
- *관심사 분리 (SoC)*: 데이터 처리 로직, 렌더링 로직, 사용자 인터랙션 로직을 명확히 분리한다
- *정보 은폐 (Information Hiding)*: 모듈 내부 구현은 숨기고 공개 인터페이스를 통해서만 상호작용한다
- *무상태 데이터 처리*: 환자 데이터를 영구 저장소에 저장하지 않고 메모리 상에서만 처리한다 (FR-5.2, HAZ-3.2)
- *외부 통신 차단*: 모든 외부 네트워크 요청을 원천 차단하여 PHI 보호를 보장한다 (FR-5.1, HAZ-3.1)
- *방어적 프로그래밍*: 비표준 입력, 메모리 부족 등 예외 상황에 대해 견고한 오류 처리를 수행한다 (NFR-3.1)

---*[SAD-05] 데이터 뷰 (Data View)*

*5.1 데이터 아키텍처 개요*

본 시스템은 환자 데이터를 영구 저장하지 않으며, 모든 데이터는 브라우저 메모리 상에서만 존재한다 (FR-5.2). 세션 종료 시 모든 데이터는 명시적으로 해제된다 (FR-5.3). 데이터베이스나 로컬 스토리지를 사용하지 않는다.

*5.2 주요 데이터 모델*

```
@startuml
skinparam backgroundColor white
skinparam shadowing false

class DICOMMetadata {
  patientName: string
  patientID: string
  studyDate: string
  modality: string
  pixelSpacing: [number, number]
  sliceThickness: number
  imageOrientationPatient: number[6]
  rows: number
  columns: number
  numberOfFrames: number
  bitsAllocated: number
  transferSyntax: string
  windowCenter: number
  windowWidth: number
}

class VolumeData {
  voxelArray: ArrayBuffer
  dimensions: [number, number, number]
  spacing: [number, number, number]
  origin: [number, number, number]
  dataType: string
  minMaxValue: [number, number]
}

class SliceData {
  imageData: ImageData
  plane: Axial | Coronal | Sagittal
  sliceIndex: number
  windowLevel: number
  windowWidth: number
}

class MeasurementData {
  startPoint: [number, number, number]
  endPoint: [number, number, number]
  distanceMM: number
  pixelSpacingValid: boolean
  disclaimerText: string
}

class ViewTransform {
  zoom: number
  panX: number
  panY: number
  windowLevel: number
  windowWidth: number
  sliceIndex: number
}

DICOMMetadata "1" -- "1" VolumeData : 파싱되어 구성
VolumeData "1" -- "*" SliceData : 단면 추출
VolumeData "1" -- "*" MeasurementData : 측정 기준
ViewTransform "1" -- "1" SliceData : 변환 적용

@enduml
```

*5.3 데이터 흐름*

1. *입력 단계*: 사용자가 File API로 DICOM 파일 선택 -> DICOMParser가 ArrayBuffer로 읽기
2. *파싱 단계*: ArrayBuffer -> DICOMMetadata + 복셀 ArrayBuffer 추출
3. *구성 단계*: 복셀 ArrayBuffer + 메타데이터 -> VolumeData 객체 생성
4. *렌더링 단계*: VolumeData + ViewTransform -> SliceData (MPR) / 3D 렌더링
5. *측정 단계*: VolumeData + 사용자 좌표 -> MeasurementData
6. *해제 단계*: 세션 종료 시 모든 ArrayBuffer 및 객체 참조 해제 -> GC

*5.4 데이터 생명주기*

||단계||저장 위치||지속 시간||비고||
|파일 로딩|브라우저 메모리 (ArrayBuffer)|세션 중만|File API로 읽은 원시 데이터|
|볼륨 데이터|브라우저 메모리 (VolumeData)|세션 중만|3D 볼륨 구성 데이터|
|렌더링 결과|Canvas 픽셀 버퍼|현재 프레임만|requestAnimationFrame 갱신|
|측정 결과|메모리 (MeasurementData)|세션 중만|화면 표시용 임시 데이터|
|세션 종료 후|없음|해제됨|캐시/스토리지 저장 금지 (FR-5.2)|

*7.2 ADR-1: Layered Architecture 채택*

- *문제 상황*: Class A 안전 등급 소프트웨어의 모듈 분리 및 관심사 분리 필요
- *대안 검토*: (1) Layered (2) Microservices (3) Event-Driven (4) Pipe-Filter
- *최종 선택*: Layered Architecture
- *근거*: 단일 브라우저 실행 환경으로 분산 불필요, DP-03 모듈식 구조 원칙 부합, 계층별 독립 테스트 가능

*7.3 ADR-2: DICOM 파서 자체 구현*

- *문제 상황*: 외부 DICOM 파서 라이브러리 의존 시 보안 감사 부담 및 외부 통신 코드 포함 위험
- *대안 검토*: (1) daikon.js (2) dicomParser (cornerstone) (3) 자체 구현
- *최종 선택*: 자체 구현
- *근거*: FR-5.4 보안 감사 요구사항, HAZ-3.1 외부 통신 차단, 의존성 최소화 전략 (SMP-02 SBOM 관리 용이)

*7.4 ADR-3: 메모리 기반 무상태 데이터 처리*

- *문제 상황*: 환자 데이터의 영구 저장으로 인한 PHI 유출 위험
- *대안 검토*: (1) IndexedDB 활용 (2) localStorage 활용 (3) 메모리 전용 처리
- *최종 선택*: 메모리 전용 처리
- *근거*: FR-5.2 캐시/스토리지 저장 금지, HAZ-3.2 데이터 잔존 방지, NFR-2.4 데이터 잔존 검증

*7.5 ADR-4: WebGL 2.0 기반 렌더링*

- *문제 상황*: 대용량 CBCT 볼륨 데이터의 실시간 렌더링 성능 확보
- *대안 검토*: (1) Canvas 2D (2) WebGL 1.0 (3) WebGL 2.0 (4) WebGPU
- *최종 선택*: WebGL 2.0
- *근거*: NFR-1.3 최소 15FPS, 3D 텍스처 지원, 브라우저 호환성(FR-6.2), WebGPU는 아직 브라우저 지원 제한적

*7.6 ADR-5: 정적 단일 번들 배포*

- *문제 상황*: 오프라인 환경에서 모든 기능 동작 보장
- *대안 검토*: (1) CDN 기반 로딩 (2) PWA + Service Worker (3) 정적 단일 번들
- *최종 선택*: 정적 단일 번들
- *근거*: FR-7.1 오프라인 전체 기능 동작, HAZ-5.3 외부 리소스 참조 제거, 단순성(Class A 요구)
