*2. [CMP-02] 형상 식별*

*2.1 형상 항목(Configuration Items) 분류 체계*
형상 관리 대상 항목은 다음과 같이 분류한다.

||분류 코드||항목 유형||예시||
|SRC|소스코드|JavaScript 모듈 (DICOM 파서, 렌더링 엔진, UI 컴포넌트, 측정 도구)|
|BLD|빌드 스크립트 및 구성 파일|package.json, vite.config.js / webpack.config.js, .eslintrc, .prettierrc|
|TST|테스트 코드 및 테스트 데이터|단위 테스트, 통합 테스트, E2E 테스트 스크립트|
|DOC|개발 산출물 문서|SDP, SRS, SAD, CMP, RMP, SMP, 테스트 보고서|
|DEP|서드파티 라이브러리 및 의존성|package-lock.json / pnpm-lock.yaml, SBOM|
|REL|릴리즈 산출물|배포 패키지, 릴리즈 노트, 검증 기록|

*2.2 명명 규칙*
- *소스코드 파일*: camelCase 명명 규칙 적용 (변수/함수), PascalCase (클래스) - DP-PLAYG-1231 섹션 3.1 참조
- *브랜치 명명 규칙*:
  - main: 프로덕션 릴리즈 브랜치
  - develop: 개발 통합 브랜치
  - feature/[JIRA-KEY]-[설명]: 기능 개발 브랜치 (예: feature/PLAYG-1001-dicom-parser)
  - release/v[MAJOR].[MINOR].[PATCH]: 릴리즈 준비 브랜치
  - hotfix/v[MAJOR].[MINOR].[PATCH]-[설명]: 긴급 수정 브랜치
- *문서 명명 규칙*: [문서유형약어]-[티켓키] (예: CMP-PLAYG-1234, DP-PLAYG-1231)
- *커밋 메시지 규칙*: [JIRA-KEY] [유형]: [설명]
  - 예: PLAYG-1001 feat: DICOM 파일 로드 기능 구현
  - 유형: feat(기능), fix(버그수정), docs(문서), refactor(리팩토링), test(테스트), chore(환경)
- *태그 명명 규칙*: v[MAJOR].[MINOR].[PATCH] (예: v1.0.0)

*2.3 버전 넘버링 규칙*
의미적 버전 관리(Semantic Versioning) 적용 (DP-PLAYG-1231 섹션 6.3 참조)
- 형식: MAJOR.MINOR.PATCH
  - MAJOR: 하위 호환성이 깨지는 변경 (예: API 변경, 아키텍처 재설계)
  - MINOR: 기능 추가 (하위 호환 유지) (예: 새로운 측정 도구 추가)
  - PATCH: 버그 수정 (하위 호환 유지) (예: DICOM 파싱 오류 수정)
- 사전 릴리즈 식별자(필요 시): v1.0.0-alpha.1, v1.0.0-beta.2
- 빌드 메타데이터(필요 시): v1.0.0+build.123
- 각 릴리즈마다 고유 버전 태그(Tag)를 부여하여 형상 추적성을 보장한다.

*3. [CMP-03] 형상 통제 절차*

*3.1 브랜치 전략*
Git Flow 기반 브랜치 전략을 채택한다 (DP-PLAYG-1231 섹션 6.1 참조).
- *main*: 프로덕션 릴리즈 브랜치. 모든 릴리즈는 main 브랜치에서 수행. 병합은 release 또는 hotfix 브랜치를 통해서만 허용
- *develop*: 개발 통합 브랜치. 기능 개발 완료 후 PR을 통해 develop에 병합. CI 파이프라인이 자동 실행됨
- *feature/*: 기능 개발 브랜치. develop에서 분기 후 완료 시 PR로 develop에 병합
  - 명명 규칙: feature/[JIRA-KEY]-[설명]
- *release/*: 릴리즈 준비 브랜치. develop에서 분기 후 버전 번호 업데이트 및 최종 테스트 수행
- *hotfix/*: 긴급 수정 브랜치. main에서 분기 후 수정 완료 시 main과 develop에 병합

*3.2 코드 리뷰 절차*
- 모든 코드 변경은 Pull Request(PR) 기반으로 관리한다.
- PR 생성 시:
  1) 작성자가 변경 내용 설명, 관련 Jira 티켓 번호, 테스트 결과를 PR 설명에 기록
  2) 자동 CI 검사 실행 (정적 분석 ESLint, 단위 테스트)
  3) 최소 1인 이상의 리뷰어 지정 (IEC 62304 Class A 요구사항)
- 리뷰 체크리스트 (DP-PLAYG-1231 섹션 3.3 참조):
  - 기능 정확성: 요구사항(SR-1 ~ SR-14) 대비 구현 일치 여부
  - 코딩 표준 준수: Airbnb/Google JavaScript Style Guide 준수 여부
  - 보안 취약점: PHI 데이터 외부 전송 여부 (SR-11 환자 정보 보호)
  - 테스트 커버리지: 변경 사항에 대한 단위 테스트 포함 여부
  - 문서 업데이트: 관련 문서 갱신 필요 여부
- 승인 조건: 모든 CI 검사 통과 + 최소 1인 리뷰어 승인(Approve)
- 병합 방식: Squash and Merge 원칙 (커밋 히스토리 정리)

*3.3 변경 승인 절차*
소프트웨어 형상 항목의 변경은 다음 승인 절차를 따른다.

||변경 유형||승인 권한||비고||
|일반 기능 개발 (feature)|코드 리뷰어 승인|PR 기반 승인|
|릴리즈 병합 (release -> main)|프로젝트 관리자 + 품질 관리자 공동 승인|DP-PLAYG-1231 섹션 5.2 참조|
|긴급 수정 (hotfix)|프로젝트 관리자 승인|사후 리뷰 필수, 위험 관리 담당자 통지|
|문서 변경|품질 관리자 승인|산출물 문서(SRS, SAD 등) 변경 시|
|빌드/배포 설정 변경|코드 리뷰어 + 형상 관리 담당자 승인|package.json, webpack/vite 설정 등|

- 변경 요청 시 관련 Jira 티켓을 반드시 연결하여 추적성을 유지한다.
- 형상 관리 담당자는 변경 이력을 기록하고 형상 상태 보고서에 반영한다.

*4. [CMP-04] 형상 상태 기록 및 보고*

*4.1 기록 항목*
형상 관리 활동과 관련하여 다음 항목을 기록하고 유지한다.
- *변경 이력*: 모든 형상 항목의 변경 내역 (Git 커밋 로그, PR 이력)
- *승인 기록*: PR 승인 내역, 승인자, 승인 일시
- *베이스라인 정보*: 각 베이스라인의 구성 형상 항목 목록 및 버전
- *결함 추적*: Jira 기반 결함(Defect) 등록, 수정, 검증 이력
- *빌드 이력*: CI/CD 파이프라인 빌드 결과, 성공/실패 기록
- *SBOM 변경 이력*: 서드파티 의존성 변경 기록 (SMP-PLAYG-1233 섹션 2.3 참조)

*4.2 보고 체계*
- *형상 상태 보고서(Configuration Status Accounting Report)*:
  - 작성 주기: Sprint 종료 시 (2주 단위)
  - 작성자: 형상 관리 담당자
  - 내용: 베이스라인 대비 변경 사항, 미해결 결함 현황, 형상 항목 버전 현황
- *릴리즈 노트(Release Notes)*:
  - 작성 시기: 각 릴리즈 배포 시
  - 내용: 버전 번호, 변경 사항, 결함 수정 내역, 알려진 제한 사항
- *정기 품질 보고*:
  - Sprint 회고 시 형상 관리 현황 공유
  - 마일스톤 달성 시 종합 형상 상태 보고 (DP-PLAYG-1231 섹션 1.3 마일스톤 참조)

*4.3 추적성 관리*
- 모든 변경 사항은 Jira 티켓과 연결되어야 한다.
- Git 커밋 메시지에 Jira 키 포함 (예: PLAYG-1001 feat: DICOM 파서 구현)
- PR 설명에 관련 Jira 티켓 링크 명시
- SyRS(SR-1 ~ SR-14) 항목과 소스코드 모듈 간 추적성 유지

*5. [CMP-05] 베이스라인 및 릴리즈 관리*

*5.1 베이스라인 정의*
베이스라인은 특정 시점의 형상 항목 집합으로, 이후 변경은 형상 통제 절차를 따라야 한다.

||베이스라인||시점||구성 형상 항목||승인 권한||
|요구사항 베이스라인|MS1: 요구사항 분석 완료|SyRS 문서, IU 문서|프로젝트 관리자 + 품질 관리자|
|설계 베이스라인|MS2: 아키텍처 설계 완료|SRS 문서, SAD 문서|프로젝트 관리자 + 품질 관리자|
|개발 베이스라인|MS3: 구현 완료|소스코드(SRC), 테스트 코드(TST), 빌드 설정(BLD)|코드 리뷰어 + 품질 관리자|
|제품 베이스라인|MS5: 검증 완료 및 배포|릴리즈 산출물(REL), 전체 문서(DOC), SBOM(DEP)|프로젝트 관리자 + 품질 관리자 공동 승인|

*5.2 베이스라인 통제 규칙*
- 베이스라인이 확정된 후의 형상 항목 변경은 형상 통제 절차(CMP-03)를 따른다.
- 베이스라인 변경 시 관련 이해관계자(프로젝트 관리자, 품질 관리자, 형상 관리 담당자)의 승인이 필요하다.
- 베이스라인 변경 이유와 영향 분석을 문서화하여 기록한다.

*5.3 릴리즈 관리*
- *릴리즈 절차* (DP-PLAYG-1231 섹션 6.4 참조):
  1) develop에서 release/v[X.Y.Z] 브랜치 분기
  2) 릴리즈 브랜치에서 통합 테스트 및 회귀 테스트 수행 (SR-1 ~ SR-14)
  3) 품질 관리자 승인 획득
  4) main 브랜치에 병합 및 버전 태그(v[X.Y.Z]) 부여
  5) 배포 아티팩트 생성 및 배포
  6) 릴리즈 노트 및 검증 기록 작성
- *배포 산출물* (DP-PLAYG-1231 섹션 6.4 참조):
  - 정적 웹 리소스 (HTML, CSS, JavaScript 번들)
  - 단일 배포 패키지 (ZIP 또는 압축 아카이브)
  - 릴리즈 노트 및 검증 기록
  - SBOM (SMP-PLAYG-1233 참조)

*5.4 긴급 릴리즈(hotfix) 절차*
- 긴급 수정이 필요한 경우 다음 절차를 따른다:
  1) main에서 hotfix/v[X.Y.Z]-[설명] 브랜치 분기
  2) 수정 구현 및 테스트
  3) 프로젝트 관리자 승인 (사후 리뷰 필수)
  4) main에 병합 및 버전 태그 부여
  5) develop에 역병합(cherry-pick 또는 merge)
  6) 위험 관리 담당자 통지 (RMP-PLAYG-1232 연계)

*6.2 Jira 메타데이터 관리*
- 모든 형상 변경은 Jira 티켓과 연결하여 관리한다.
- Jira 티켓 필드를 활용한 메타데이터 관리:
  - 이슈 유형: Bug(결함), Task(작업), Story(기능) 등으로 분류
  - 우선순위: 변경의 긴급도 및 영향도에 따라 설정
  - 버전 필드: 영향을 받는 버전(Affects Version) 및 수정 버전(Fix Version) 관리
  - 컴포넌트: 기능 모듈별 분류 (DICOM 파서, 렌더링 엔진, UI, 측정 도구 등)
  - 링크: 관련 티켓 간 연결 (형상 항목 추적성 확보)

*6.3 CI/CD 파이프라인*
- *CI (지속적 통합)*:
  - 트리거: Pull Request 생성/업데이트 시 자동 실행
  - 단계: 코드 체크아웃 -> 의존성 설치 -> 정적 분석(ESLint) -> 단위 테스트 -> 빌드 검증
  - 기준: 모든 단계 통과 시에만 PR 병합 허용
- *CD (지속적 배포)*:
  - 트리거: main 브랜치 병합 시 자동 실행
  - 단계: 코드 체크아웃 -> 의존성 설치 -> 프로덕션 빌드 -> 아티팩트 생성
  - 산출물: 정적 웹 리소스 번들, 배포 패키지

*6.4 백업 및 복구*
- 원격 저장소(GitHub/GitLab)를 통한 소스코드 백업 보장 (DP-PLAYG-1231 섹션 6.6 참조)
- 산출물 문서는 형상 관리 저장소 내 docs/ 디렉토리에 관리
- 정기적 백업 검증 수행
- 저장소 접근 권한은 최소 권한 원칙에 따라 관리

*6.5 보안 관리*
- 저장소 접근 제어: 역할 기반 접근 권한 관리
- 브랜치 보호: main, develop 브랜치는 직접 push 금지, PR을 통해서만 변경
- 시크릿 관리: API 키, 인증서 등 민감 정보는 저장소에 평문으로 저장하지 않음
- 본 제품은 오프라인 로컬 동작(SR-9)을 원칙으로 하므로 네트워크 관련 보안 위험은 제한적 (SR-11 참조)
