*1. [RMP-01] 목적 및 적용 범위*

*1.1 목적*
본 문서는 Simple CBCT Viewer 소프트웨어의 위험 관리 활동을 계획하고 관리하기 위한 위험 관리 계획서(Risk Management Plan)이다. ISO 14971:2019 및 IEC 62304:2006+AMD1:2015 제7절 요구사항에 따라 위험 분석, 평가, 통제 및 모니터링 활동을 정의한다.

*1.2 적용 대상 소프트웨어*
- 제품명: Simple CBCT Viewer
- 유형: 웹 브라우저 기반 CBCT 영상 뷰어 (독립형 소프트웨어)
- 의도된 목적: 치과용 CBCT 영상의 조회 및 보조 분석 (진단 목적 아님)
- Intended Use 참조: PLAYG-1237 (CBCT Viewer - Intended Use Statement)

*1.3 수명 주기 단계*
본 위험 관리 계획은 다음 소프트웨어 수명 주기 단계 전체에 적용된다.
- 요구사항 분석 단계 (SyRS 작성, PLAYG-1229 참조)
- 아키텍처 설계 단계 (SAD 작성)
- 상세 설계 및 구현 단계
- 검증(Verification) 및 확인(Validation) 단계
- 배포 및 출시 단계
- 출시 후 모니터링 단계

*1.4 관련 규격 및 참조 문서*
- ISO 14971:2019 (의료기기 - 위험 관리의 응용)
- IEC 62304:2006+AMD1:2015 (의료기기 소프트웨어 - 소프트웨어 수명 주기 프로세스)
- IEC 62304 안전 등급: Class A (PLAYG-1290 Classification 근거)
- Intended Use: PLAYG-1237
- System Requirement Specification: PLAYG-1229 (SyRS SR-1 ~ SR-12)
- Software Development Plan: PLAYG-1231 (DP-PLAYG-1231)
- Security Maintenance Plan: PLAYG-1233
- Configuration Management Plan: PLAYG-1234

*2. [RMP-02] 위험 관리 프로세스 개요*

본 프로젝트의 위험 관리 프로세스는 ISO 14971:2019의 요구사항에 따라 다음 4단계로 구성된다.

*2.1 위험 식별 (Risk Identification)*
- SyRS 각 항목(SR-1 ~ SR-12) 및 Intended Use(PLAYG-1237)를 기반으로 잠재적 위험원(Hazard)을 식별
- 위험원 식별 기법: FMEA(실패 유형 및 영향 분석)
- 식별 대상: 정상 사용 조건 및 합리적으로 예측 가능한 오사용
- 주요 위험 영역 (DP-PLAYG-1231 섹션 4.2 참조):
  - 영상 렌더링 왜곡: DICOM 파싱 또는 렌더링 오류로 인한 영상 왜곡 가능성 (SR-1, SR-2)
  - 거리 측정 오류: Pixel Spacing 해석 오류로 인한 측정 부정확 (SR-6)
  - 환자 정보 유출: DICOM 파일 내 PHI 데이터의 외부 전송 가능성 (SR-10)
  - 경고 문구 누락: 사용자 제약 경고가 표시되지 않을 가능성 (SR-9)
  - 브라우저 호환성 문제: 특정 브라우저에서 기능 동작 불량 (SR-8)

*2.2 위험 분석 및 평가 (Risk Analysis and Evaluation)*
- 식별된 각 위험원에 대해 위해 사건(Hazardous Situation)을 정의
- 각 위해 사건에 대해 Severity(심각도)와 Probability(발생 확률)를 평가
- Risk = Severity x Probability 매트릭스를 사용하여 위험 수준 산출
- 산출된 위험 수준을 RMP-03의 수용 기준과 비교하여 수용 가능성 판단

*2.3 위험 통제 (Risk Control)*
- 수용 불가능한 위험에 대해 위험 통제 조치를 수립
- 통제 조치 우선순위 (ISO 14971 제거 순서):
  1) 본질적 안전 설계 (Inherent Safety by Design)
  2) 보호 조치 (Protective Measures)
  3) 정보 전달 (Information for Safety)
- 통제 조치 구현 후 잔여 위험을 재평가

*2.4 잔여 위험 검토 (Residual Risk Review)*
- 모든 위험 통제 조치 적용 후 잔여 위험을 종합적으로 검토
- 개별 잔여 위험이 수용 기준을 충족하는지 확인
- 전체 잔여 위험의 누적 효과를 평가하여 전반적 잔여 위험 수용 여부 판단
- 위험-편익 분석(Risk-Benefit Analysis) 수행 (필요 시)

*2.5 프로세스 흐름*
위험 식별 -> 위해 사건 정의 -> 위험 분석(Severity/Probability 평가) -> 위험 평가(수용 기준 비교) -> 위험 통제 조치 수립 -> 통제 조치 구현 -> 잔여 위험 재평가 -> 전반적 잔여 위험 검토 -> 위험 관리 보고서 작성

*3. [RMP-03] 위험 수용 기준*

*3.1 Severity(심각도) 분류*

||등급||설명||예시 (CBCT Viewer 기준)||
|S1 - 무시 가능|환자 또는 사용자에게 영향 없음|UI 사소한 표시 오류 (측정/영상 무관)|
|S2 - 경미|일시적 불편, 치료에 영향 없음|영상 로딩 지연, 사소한 UI 오류|
|S3 - 중간|임상적 의사결정에 잠재적 영향|영상 명암 왜곡, 측정값 경미한 오차|
|S4 - 중대|환자 안전에 간접적 영향 가능|거리 측정 상당 오차, 경고 문구 누락|
|S5 - 치명|환자 안전에 직접적 위해 가능|영상 완전 왜곡으로 오진 유도|

*3.2 Probability(발생 확률) 분류*

||등급||설명||근사 확률||
|P1 - 극히 낮|사실상 발생하지 않음|< 0.01%|
|P2 - 낮|드물게 발생|0.01% ~ 0.1%|
|P3 - 보통|간혹 발생|0.1% ~ 1%|
|P4 - 높음|종종 발생|1% ~ 10%|
|P5 - 매우 높음|빈번하게 발생|> 10%|

*3.3 위험 매트릭스 (Risk Matrix)*

|| ||P1 극히 낮||P2 낮||P3 보통||P4 높음||P5 매우 높음||
|S5 치명|중간|높음|높음|높음|높음|
|S4 중대|낮음|중간|높음|높음|높음|
|S3 중간|낮음|낮음|중간|중간|높음|
|S2 경미|수용|낮음|낮음|낮음|중간|
|S1 무시 가능|수용|수용|수용|낮음|낮음|

*3.4 위험 수준별 조치*
- *수용 (Acceptable)*: 추가 조치 불필요. 모니터링만 수행.
- *낮음 (Low)*: ALARP 원칙 적용. 합리적으로 실현 가능한 통제 조치 구현.
- *중간 (Medium)*: 위험 통제 조치 필수. 통제 후 잔여 위험 재평가.
- *높음 (High)*: 즉각적 위험 통제 조치 필수. 위험-편익 분석 필요. 통제 후에도 높음 시 설계 변경.

*3.5 Class A 소프트웨어 위험 수용 기준*
본 소프트웨어는 IEC 62304 Class A로 분류(PLAYG-1290)되므로, 소프트웨어로 인한 위험이 낮음(Low) 이하인 경우 수용 가능하다. 중간(Medium) 이상의 위험은 반드시 통제 조치를 구현해야 하며, ALARP 원칙에 따라 합리적으로 실현 가능한 모든 통제 조치를 적용한다.

*4. [RMP-04] 역할 및 책임*

*4.1 위험 관리 조직*

||역할||이름/직책||주요 책임||
|위험 관리 책임자 (Risk Manager)|품질 관리자|위험 관리 활동 총괄, 위험 평가 승인, 잔여 위험 수용 판단, 위험 관리 보고서 승인|
|소프트웨어 개발팀|개발자|위험원 식별 지원, 위험 통제 조치 구현, 단위/통합 테스트를 통한 통제 조치 검증|
|QA/테스트 엔지니어|QA 엔지니어|위험 기반 테스트 수행, 통제 조치 검증, 결함 추적|
|위험 관리 리뷰어|프로젝트 관리자|위험 평가 결과 독립 검토, 위험 관리 활동 감사|

*4.2 승인 권한*
- 위험 평가 및 수용: 위험 관리 책임자(품질 관리자)
- 위험 통제 조치 구현: 소프트웨어 개발팀 -> 코드 리뷰어 검증
- 전반적 잔여 위험 수용: 위험 관리 책임자 + 프로젝트 관리자 공동 승인
- 위험 관리 보고서 최종 승인: 품질 관리자

*4.3 독립성 요구사항*
- 위험 평가 및 수용 판단은 개발팀과 독립된 품질 관리자가 수행
- IEC 62304 Class A 요구사항에 따라 최소한의 독립성을 보장

*5. [RMP-05] 위험 관리 활동 계획*

*5.1 초기 위험 분석 (요구사항 단계)*
- *시기*: SyRS 작성 완료 후 (MS1 마일스톤)
- *활동*:
  - SyRS(SR-1 ~ SR-12) 각 항목별 잠재적 위험원 식별
  - Intended Use(PLAYG-1237) 기반 위험원 식별
  - 초기 Severity/Probability 평가 및 위험 매트릭스 적용
  - 위험 통제 요구사항을 설계 입력으로 전환
- *산출물*: 초기 위험 분석 보고서

*5.2 설계/구현 단계 지속 위험 관리*
- *시기*: 아키텍처 설계(MS2) ~ 구현 완료(MS3)
- *활동*:
  - 아키텍처 설계 검토 시 위험 완화 조치 반영 여부 확인
  - 상세 설계 단계에서 위험 통제 조치의 구현 방법 정의
  - 구현 단계에서 코드 레벨 위험 통제 조치 적용:
    - 입력 데이터 검증: DICOM 파일 유효성 검사 및 예외 처리 (SR-1, SR-2)
    - 메타데이터 검증: Pixel Spacing 등 측정 메타데이터 유효성 확인 (SR-6)
    - 외부 전송 차단: 네트워크 요청 코드 레벨 차단 및 정적 분석 검증 (SR-10)
    - UI 경고 표시: 진단 불가 경고 문구 의무 표시 (SR-9)
  - 위험 통제 조치별 단위 테스트 작성 및 실행
- *산출물*: 갱신된 위험 등록부

*5.3 검증 단계 위험 관리*
- *시기*: 단위/통합/시스템 테스트 (MS4)
- *활동*:
  - 식별된 위험 항목에 대한 검증 테스트 수행
  - 위험 통제 조치의 실효성 검증:
    - 크로스 브라우저 테스트: Chrome, Edge, Firefox (SR-8)
    - 측정 정확도 테스트: 거리 측정 오차 범위 확인 (SR-6)
    - 오프라인 동작 테스트: 네트워크 차단 상태에서 기능 확인 (SR-7)
    - 보안 테스트: 외부 데이터 전송 여부 정적 분석 (SR-10)
  - 검증 결과 기반 잔여 위험 재평가
- *산출물*: 검증 테스트 결과, 갱신된 위험 등록부

*5.4 출시 후 모니터링 (Post-Market Monitoring)*
- *시기*: 제품 출시 후 지속
- *활동*:
  - 사용자 피드백 및 불만 사항 수집 체계 운영
  - 식별된 새로운 위험원에 대한 위험 분석 수행
  - 기존 위험 평가의 적절성 정기 재검토
  - 필요 시 위험 관리 계획 업데이트 및 추가 통제 조치 구현
  - 생산 후 모니터링 데이터를 다음 버전 위험 분석에 반영
- *산출물*: 생산 후 모니터링 보고서, 위험 관리 계획 개정판 (필요 시)

*6. [RMP-06] 위험 관리 문서화 및 추적성*

*6.1 위험 관리 문서 체계*

||문서||위치(참조)||갱신 시기||
|위험 관리 계획서 (RMP)|본 문서 (PLAYG-1232)|프로젝트 시작 시 작성, 주요 변경 시 갱신|
|위험 등록부 (Risk Register)|Jira Hazard 티켓|위험 식별 시 즉시 생성, 통제 조치 구현 시 갱신|
|위험 관리 보고서|RMR 문서|모든 위험 관리 활동 완료 후 작성|
|FMEA 분석서|Hazard 티켓 설명 필드|위험 분석 수행 시 작성|
|위험 로그|Hazard 티켓 댓글/기록|위험 상태 변경 시 갱신|

*6.2 위험 등록부 관리 (Jira Hazard 티켓)*
- 각 식별된 위험원은 Jira Hazard 티켓으로 생성하여 관리
- Hazard 티켓 필드:
  - 위험원(Hazard) 명칭
  - 위해 사건(Hazardous Situation) 설명
  - Severity 등급 (S1 ~ S5)
  - Probability 등급 (P1 ~ P5)
  - 초기 위험 수준 (Risk Level)
  - 위험 통제 조치 (Risk Control Measures)
  - 잔여 위험 수준 (Residual Risk Level)
  - 통제 조치 검증 결과
  - 상태 (Open / In Control / Closed)

*6.3 추적성 (Traceability)*
- 각 Hazard 티켓은 다음 항목과 연결(LInk)하여 추적성을 보장:
  - SyRS 요구사항 티켓 (PLAYG-1293 ~ PLAYG-1306)
  - Intended Use 티켓 (PLAYG-1237)
  - 소프트웨어 설계 요소 (SAD 모듈)
  - 테스트 케이스 / 테스트 결과
  - 위험 통제 관련 소스코드 모듈

*6.4 문서 갱신 주기*
- 위험 관리 계획서(RMP): 프로젝트 시작 시 작성, 주요 설계 변경 또는 새로운 위험원 식별 시 갱신
- 위험 등록부(Hazard 티켓): 위험 식별 시 즉시 생성, 각 개발 단계(MS1~MS5)마다 재검토 및 갱신
- 위험 관리 보고서(RMR): 모든 위험 관리 활동 완료 후, 제품 출시 전 최종 작성
- FMEA 분석서: 요구사항 변경 또는 설계 변경 시 갱신

*6.5 승인 및 보관*
- 모든 위험 관리 문서는 품질 관리자의 승인을 득한 후 공식 문서로 확정
- 확정된 문서는 형상 관리 저장소 내 docs/ 디렉토리에 보관 (PLAYG-1234 CMP 참조)
